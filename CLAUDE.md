# CLAUDE.md

## 项目概述

基于 Python 3.13+ 的 RAG（检索增强生成）项目，包管理使用uv。

## 开发规范

### 0. 测试框架

本项目使用 **pytest** 作为测试框架。测试文件放在 `tests/` 目录下，文件命名遵循 `test_*.py` 规范。

### 1. 先理解，再方案，后确认

收到任何开发需求后，必须遵循以下流程：

1. **理解需求**：仔细分析需求，确保完全理解意图。
2. **整理方案**：梳理实现思路，包括涉及的文件、改动范围、技术方案。
3. **输出方案**：用简洁清晰的语言向用户说明大致的处理思路。
4. **等待确认**：**必须在用户明确同意方案后**，才能开始编写代码。

**禁止**在未获得用户确认的情况下直接开始编码。

### 2. 需求不明必须及时确认

如果需求存在以下情况，**必须先向用户提问澄清**：

- 描述模糊或有歧义
- 信息不完整，缺少关键细节
- 存在多种可能的理解方式
- 边界情况未明确

**禁止**自行假设或猜测用户意图后直接执行。

### 3. 新功能必须创建单元测试

每次开发新功能，必须同步创建对应的单元测试：

- 测试文件放在 `tests/` 目录下
- 测试函数命名清晰，反映测试场景
- 覆盖正常路径和关键边界情况

### 4. 修改后必须运行单元测试

每次功能修改完成后，必须运行 pytest 验证：

```bash
python -m pytest tests/ -v
```

- 确保全部测试通过
- 如有测试失败，必须先修复再通知用户
- 如果有前端页面的情况下，请你启动MCP的浏览器插件，测试系统正常运行后，再提醒我使用系统。

### 5. 日志记录模块

开发项目过程中，一定要写日志。日志需要既输出到控制台，也要输出到文件中。每次处理程序问题的时候，需要依据日志的报错，修复bug。

### 6. 测试通过后再通知验收

只有所有单元测试通过后，才能通知用户进行检查验收。通知时应简要说明：

- 修改了哪些内容
- 测试结果（全部通过）

如果测试未通过，不要通知用户验收，先定位并修复问题。
